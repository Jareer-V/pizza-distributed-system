تقرير مشروع PizzaChain: نظام طلبات بيتزا موزع

هذا التقرير يشرح كيف يعمل المشروع، وما هي الطرق الموزعة التي تم تطبيقها، وكيفية اختبار العرض التوضيحي واستخداماته العملية مع أمثلة رقمية واضحة.

1. مقدمة

مشروع PizzaChain هو نموذج تقني يوضح كيف يمكن بناء نظام طلبات للمطاعم أو المتاجر باستخدام بنية خدمات صغيرة موزعة. الهدف الرئيسي هو أن العميل يرسل طلبًا واحدًا فقط، بينما يتولى النظام الداخلي تقسيم العمل بين عدة خدمات مثل: بوابة الدخول، خدمة المخزون، خدمة الدفع، وخدمة التوصيل.

الفكرة الأساسية هي أن العميل لا يحتاج إلى معرفة أين توجد الخدمات أو كيف تتواصل مع بعضها. هذا ما يسمى بالشفافية في التصميم الموزع.

2. كيف يعمل المشروع

الخطوات الأساسية

  • العميل يرسل طلبًا عبر الواجهة الأمامية.
  • البوابة تستقبل الطلب من نقطة دخول واحدة.
  • البوابة تتواصل مع خدمة Kitchen للتحضير.
  • ثم تتواصل مع خدمة Billing للدفع.
  • وأخيرًا مع خدمة Delivery للتوصيل.

مثال عملي

إذا قدم العميل طلبًا لبيتزا مارغريتا بكمية 2 وسعر 12.99 دولار لكل قطعة، فإن النظام يحسب القيمة الكلية كالتالي:

2 × 12.99 = 25.98

ثم يرسل الطلب إلى الخدمات الداخلية ويعيد النتيجة النهائية كطلب مؤكد.

3. الطرق الموزعة التي يعمل بها المشروع

3.1 طبقة البوابة (API Gateway)

البوابة هي نقطة الدخول الواحدة إلى النظام. بدلاً من أن يتعامل العميل مباشرة مع خدمات كثيرة، يرسل الطلب إلى بوابة واحدة فقط. هذه البوابة تقوم بتنظيم التدفق وإخفاء تفاصيل الخدمات الداخلية.

الفائدة: تبسيط الاتصال، تحسين الأمان، وسهولة التحكم في الطلبات.

3.2 gRPC مع بروتوكول Protobuf

تم استخدام gRPC بين المكونات التي تحتاج سرعة عالية وتواصلًا منظمًا، خاصةً في خدمة المخزون. البروتوكول الثنائي أسرع وأكثر كفاءة من JSON في بعض الحالات.

الفائدة: سرعة أعلى، ثبات في الهيكل، وتوافق جيد بين الخدمات.

3.3 موازن التحميل Least Connections

عندما يتعرض النظام لحركة مرور كثيرة، يتم اختيار الخادم الذي لديه أقل عدد من الاتصالات الحالية. مثال: إذا كان هناك 3 خوادم بحالات 4، 1، 7 اتصالات، فإن النظام يختار الخادم الذي يحتوي على 1 اتصال فقط.

الفائدة: توزيع الطلبات بشكل عادل وتقليل الضغط على خادم واحد.

3.4 Idempotency

تمت إضافة مفتاح Idempotency Key لمنع تكرار العملية عند إعادة إرسال الطلب نفسه. وهذا مهم جدًا في حالة فشل الشبكة أو التكرار غير المقصود.

مثال: إذا أرسل العميل نفس الطلب مرتين، فسيتم التعامل معه كطلب واحد بدلًا من فرض رسوم مرتين.

3.5 Circuit Breaker

إذا فشلت خدمة الدفع أو أصبحت بطيئة جدًا، يتدخل Circuit Breaker ويوقف إرسال الطلبات إلى تلك الخدمة مؤقتًا لتفادي التراكم والشلل. بعد 5 فشلٍ متتالي، يتم فتح الدائرة.

الفائدة: حماية النظام من الأعطال المتتالية وتحسين الاستقرار.

3.6 Sharding

تم تقسيم البيانات إلى أقسام أو شظايا (Shards) حسب المعرف أو المنطقة الجغرافية. على سبيل المثال، يمكن توزيع الطلبات على شظايا مثل ALPHA وBETA وGAMMA بناءً على قيمة Order ID.

مثال: إذا كان Order ID = 2,000,000، فسيتم توجيهه إلى شارد BETA لأن قيمته تقع في النطاق من 1,000,000 إلى 5,000,000.

4. كيف يتم اختبار العرض التوضيحي

4.1 التشغيل محليًا

  1. تنفيذ npm install
  2. تشغيل الخدمات الوهمية: npm run mock-services
  3. تشغيل البوابة: npm run gateway
  4. فتح صفحة الواجهة الأمامية في المتصفح.
  5. ضغط زر Place Order.

4.2 اختبار الطلب عبر API

يمكن اختبار النظام مباشرة عبر طلب HTTP إلى نقطة الدخول:

POST /api/v1/order

مثال JSON:

{"customerId":"cust-demo-001","items":[{"productId":"PIZZA_MARGHERITA","name":"Margherita Pizza","qty":1,"price":12.99}],"deliveryAddress":"42 Baker Street, London","paymentMethod":"card","idempotencyKey":"demo-key-001"}

النتيجة المتوقعة هي استجابة بـ 201 Created مع Order ID وTracking ID وTransaction ID.

4.3 ما الذي يجب ملاحظته عند الاختبار؟

  • أن الطلب يُقبل بنجاح.
  • أن الخدمة تعيد حالة CONFIRMED.
  • أن لا يحدث دفع مزدوج عند إعادة الإرسال نفسه.
  • أن البوابة تُظهر Trace ID أو Request ID لمتابعة الطلب.
5. حالات الاستخدام العملية

الحالة 1: طلب بيتزا عادي

عميل يطلب 1 بيتزا مارغريتا بسعر 12.99 دولار، ويختار الدفع بالبطاقة. النظام ينجح في التحضير والدفع والتوصيل، ويعيد وقت تقديري للتوصيل يبلغ 30 دقيقة.

الحالة 2: ذروة الحركة

إذا وصل عدد الطلبات إلى 100 طلب/دقيقة، فسيتم توزيعها على الخوادم بناءً على أقل عدد اتصالات. هذا يقلل من التكدس ويحسن الاستجابة.

الحالة 3: إعادة إرسال الطلب

إذا فشل الطلب الشبكي بعد الإرسال، فإن مفتاح Idempotency Key يمنع فرض رسوم مرتين. هذا مهم جدًا في الأنظمة التجارية.

الحالة 4: فشل الخدمة

إذا تعطلت خدمة الدفع، سيبدأ Circuit Breaker بالحماية. بدلًا من استمرار محاولات الفشل، يمر الطلب إلى حالة بديلة أو يُظهر فشلًا واضحًا للمستخدم.

الحالة 5: البيانات الكبيرة

عند وجود ملايين الطلبات، يعتمد النظام على Sharding لتوزيع البيانات بين أكثر من شارد، ما يحسن الأداء ويجعل الاستعلامات أسرع.

6. الخلاصة

مشروع PizzaChain يُظهر كيف يمكن تطبيق مفاهيم الأنظمة الموزعة عمليًا باستخدام بنية بسيطة لكن واقعية. المشروع لا يركز فقط على واجهة المستخدم، بل يوضح أيضًا كيف يمكن تصميم نظام يدعم التوسع، المرونة، والحماية من الأعطال باستخدام أدوات مثل البوابة، gRPC، موازن التحميل، Idempotency، Circuit Breaker، وSharding.

هذا يجعل المشروع مناسبًا جدًا كعرض عملي لمادة الأنظمة الموزعة أو هندسة البرمجيات.
